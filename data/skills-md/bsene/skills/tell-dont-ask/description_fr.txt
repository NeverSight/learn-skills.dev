Appliquez le principe Tell Don't Ask (TDA) lors de la révision, de l'écriture ou de la refactorisation du code orienté objet. Utilisez cette compétence chaque fois que l'utilisateur pose des questions sur la conception POO, mentionne les getters/setters, souhaite examiner une classe pour des problèmes d'encapsulation, demande comment rapprocher la logique des données ou demande pourquoi le code semble « procédural » malgré l'utilisation de classes. Se déclenche également lorsque l'utilisateur demande de refactoriser le code qui interroge l'état d'un objet avant de prendre des décisions en externe. Cette compétence devrait être utile pour toute révision de code ou question de conception impliquant des modèles d'accès aux données, l'encapsulation ou la manière dont les objets doivent collaborer.

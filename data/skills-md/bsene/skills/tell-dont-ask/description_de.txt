Wenden Sie das Tell Don't Ask (TDA)-Prinzip an, wenn Sie objektorientierten Code überprüfen, schreiben oder umgestalten. Verwenden Sie diese Fähigkeit immer dann, wenn der Benutzer Fragen zum OOP-Design stellt, Getter/Setter erwähnt, eine Klasse auf Kapselungsprobleme überprüfen möchte, fragt, wie die Logik näher an die Daten herangeführt werden kann, oder fragt, warum sich Code trotz der Verwendung von Klassen „prozedural“ anfühlt. Wird auch ausgelöst, wenn der Benutzer eine Umgestaltung des Codes anfordert, der den Status eines Objekts abfragt, bevor er extern Entscheidungen trifft. Diese Fähigkeit sollte bei jeder Codeüberprüfung oder Designfrage zum Einsatz kommen, die Datenzugriffsmuster, Kapselung oder die Art und Weise, wie Objekte zusammenarbeiten sollen, betrifft.

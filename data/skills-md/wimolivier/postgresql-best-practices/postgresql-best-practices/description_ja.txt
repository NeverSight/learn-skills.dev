PostgreSQL 18 以降のエンタープライズ データベース開発のための包括的なベスト プラクティス。スキーマを提供します
アーキテクチャ パターン、テーブル API 設計、PL/pgSQL コーディング標準、移行、データ ウェアハウジング。

ユーザーが次の場合にこのスキルを使用します。
- PostgreSQL スキーマ、テーブル、関数、プロシージャ、またはトリガーを作成します。
- PostgreSQL のデータ型 (uuid、text、timestamptz、jsonb、numeric) について質問します。
- PL/pgSQL コードを記述し、命名規則 (l_、in_、io_、co_ 接頭辞) が必要です
- テーブル API パターンの実装 (SECURITY DEFINER 関数、スキーマ分離)
- データベースの移行またはスキーマのバージョン管理をセットアップします。
- インデックスの最適化、制約設計、またはクエリ パフォーマンスのサポートが必要
- PostgreSQL 18 以降の機能 (uuidv7、仮想列、時間的制約) について質問します。
- メダリオン アーキテクチャ (ブロンズ/シルバー/ゴールド) でデータ ウェアハウスを構築
- データリネージ追跡、ETL パターン、または監査ログが必要
- ベスト プラクティスまたはアンチパターンについてデータベース コードをレビューします。
- Oracle PL/SQL から PostgreSQL PL/pgSQL への移行
- データベース変更用の CI/CD パイプラインをセットアップします。

コアパターン:
- 3 つのスキーマの分離: データ (テーブル) → プライベート (内部) → API (外部)
- テーブル API: SET search_path を使用した SECURITY DEFINER 関数を介したすべてのアクセス
- ネイティブ移行システム: Flyway/Liquibase に代わる純粋な PL/pgSQL
- Trivadis 命名: l_ (ローカル)、in_ (入力)、io_ (inout)、co_ (定数)

Helps decide whether to wrap primitive types in domain-specific types. Avoids the extremes of Primitive Obsession (overusing primitives) and Value Object Obsession (over-wrapping), offering rational criteria based on return on investment. Also prevents terminology confusion caused by differing definitions of Value Object across contexts (PofEAA, DDD, general). Use when deciding whether to wrap primitives during code reviews, new implementations, or design discussions. Target languages: language-agnostic (Rust, TypeScript, Java, Kotlin, Scala, Go, Python, etc.). Triggered by requests related to wrapping primitives, such as “Should I wrap this value?”, “Is it okay to leave it as a primitive?”, “Should it be a Value Object?”, “Am I creating too many types?”, “Primitive Obsession”, “Over-wrapping”, “Too many types”, “Is String fine as-is?”

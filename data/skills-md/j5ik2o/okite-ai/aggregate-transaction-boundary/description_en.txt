Clarify the relationship between aggregates and transaction boundaries, and detect/correct the anti-pattern of including multiple aggregates in a single transaction. Apply the principle that an aggregate is a strong consistency boundary, so when multiple aggregates must be updated in a use-case, use eventual consistency. Use during code reviews, use-case design, or refactoring to spot transaction-boundary issues. Target languages: language-agnostic (Java, Kotlin, Scala, TypeScript, Go, Rust, Python, etc.). Triggers: requests about transaction boundaries such as “updating multiple aggregates in the same transaction,” “@Transactional on the use-case,” “how to keep consistency across aggregates,” “should I use the Saga pattern,” “designing transaction boundaries,” “one transaction per aggregate,” “implementing eventual consistency,” or “transactions spanning aggregates.”

Clarify the relationship between aggregates and transaction boundaries, detect and correct the anti-pattern of including multiple aggregates in a single transaction. Apply the principle that an aggregate is a strong consistency boundary and, when multiple aggregates must be updated in a use-case, eventual consistency should be used. Employ during code reviews, use-case design, and refactoring to spot transaction-boundary issues. Target languages: language-agnostic (Java, Kotlin, Scala, TypeScript, Go, Rust, Python, etc.). Triggers: requests about transaction boundaries such as “updating multiple aggregates in the same transaction,” “@Transactional on a use-case,” “how to ensure consistency across aggregates,” “should I use the Saga pattern,” “designing transaction boundaries,” “one transaction per aggregate,” “implementing eventual consistency,” “transactions spanning aggregates.”
